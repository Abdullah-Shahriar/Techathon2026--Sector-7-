# OfficePulse AI Telemetry Contract

This is the shared contract used by the simulator and future physical ESP32 room nodes. The backend must not need to know which sender produced a valid payload.

## Transport

```text
Method: POST
Path: /api/iot/telemetry
Content-Type: application/json
x-device-api-key: <shared device secret>
Success: HTTP 202
```

Only this ingestion endpoint requires `x-device-api-key`. The simulator and backend must use the same `DEVICE_API_KEY`.

## Payload Version 1.0

```json
{
  "schemaVersion": "1.0",
  "nodeId": "room-node-work1",
  "sequence": 42,
  "eventType": "state_change",
  "changedDeviceIds": ["work1-fan-1"],
  "devices": [
    {
      "id": "work1-fan-1",
      "status": "on",
      "measurements": {
        "voltageVolts": 220,
        "currentAmps": 0.273,
        "powerWatts": 60
      }
    }
  ]
}
```

## Field Rules

| Field | Type | Rules |
| --- | --- | --- |
| `schemaVersion` | string | Must be exactly `"1.0"` |
| `nodeId` | string | Non-empty stable ESP32/node identifier |
| `sequence` | integer | Non-negative and increasing per node |
| `eventType` | enum | `boot`, `heartbeat`, `state_change`, or `manual_sync` |
| `changedDeviceIds` | string[] | IDs changed by the event; empty for full snapshots when appropriate |
| `devices` | array | At least one device; each object is strict |
| `devices[].id` | string | Non-empty stable device identifier |
| `devices[].status` | enum | `on` or `off` |
| `measurements.voltageVolts` | number | Non-negative |
| `measurements.currentAmps` | number | Non-negative |
| `measurements.powerWatts` | number | Non-negative |

Unknown top-level, device, or measurement fields are rejected because the Zod schemas are strict.

## Event Meanings

| Event | Expected use |
| --- | --- |
| `boot` | Initial full node snapshot after startup/reconnect |
| `heartbeat` | Periodic full snapshot proving node health |
| `state_change` | Immediate update after one or more physical device changes |
| `manual_sync` | Explicit operator-requested snapshot |

## Sequence Handling

- The backend records every valid request in `telemetry_events`.
- A newer sequence can update node health, latest device state, usage intervals, and alerts.
- Duplicate or older sequences are recorded in `telemetry_events` and `node_sequence_logs` but do not mutate live state, usage, or alerts.
- A sequence gap is recorded as `missed` and may create a `missed_telemetry_sequence` alert.
- Sequence ownership belongs to each node; restart behavior should preserve or deliberately reinitialize it with a `boot` event.

## Sender Responsibilities

The ESP32 or simulator owns:

- Stable `nodeId`, device IDs, sequence number, and event type
- Current `on`/`off` state
- Current voltage, current, and power measurements
- Immediate state-change sends and periodic heartbeats

The sender must not send:

- Room database IDs or room names
- Device/room/office kWh or BDT cost
- Office-time or off-time classification
- Alert decisions
- Backend timestamps, on-duration, or monthly estimates
- Simulator-only profiles, rated-wattage controls, GPIO metadata, or source labels

## Backend Responsibilities

The backend:

- Authenticates and validates telemetry
- Stamps authoritative receive time
- Discovers unknown nodes/devices
- Maps nodes and devices to rooms
- Opens/closes usage intervals and integrates kWh
- Calculates BDT cost and office/off-time totals
- Evaluates device, node, room, and office alerts
- Broadcasts Socket.IO updates

## Response Envelope

Accepted telemetry:

```json
{
  "ok": true,
  "data": {
    "accepted": true
  }
}
```

The data object also reports ingestion/discovery details generated by the service.

Invalid API key:

```json
{
  "ok": false,
  "error": {
    "message": "Invalid or missing device API key"
  }
}
```

Invalid payloads return HTTP `400` and are retained as invalid telemetry events when possible.

## Current Simulator Profile

The included simulator emits three room nodes with five devices each:

| Room | Node | Devices |
| --- | --- | --- |
| Drawing Room | `room-node-drawing` | 2 fans, 3 lights |
| Work Room 1 | `room-node-work1` | 2 fans, 3 lights |
| Work Room 2 | `room-node-work2` | 2 fans, 3 lights |

Default fan power is `60 W`, default light power is `15 W`, and off devices report zero voltage/current/power. The backend itself is not fixed to this demo count and can discover additional node and device IDs.
